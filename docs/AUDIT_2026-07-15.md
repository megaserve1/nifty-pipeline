# Pipeline audit — 2026-07-15

How it was done: ten reviewers went through every module in parallel, each allowed to
write and RUN its own synthetic test cases against the real code. One of them executed
the entire chain (register → freeze → build → certify → train-shape) on synthetic data.
89 findings came back; each was then re-verified against the code before anything was
changed. Everything below was fixed the same day, and the full test suite
(**177 tests**) passes.

---

## The three critical bugs (all fixed, all verified by executed repro)

**1. `import config` crashed any machine without the labels CSV.**
The labels path was resolved at import time, and every trainer and core script imports
config. A ClearML agent on a fresh VM clone — which needs the labels for nothing —
died before it could fetch a single byte. Now resolved lazily; only
`build_dataset.py` ever asks for the file.

**2. ClearML never received the trainers' arguments — every queued run would have
"trained" nothing, green.**
ClearML hooks argparse by patching it AT IMPORT. All three trainer scripts imported
clearml AFTER parsing, so the parse was invisible: a cloned task's `Args/dataset_id`
never arrived, the script saw no dataset id, printed "registration run" and exited
cleanly. All three models. No error anywhere. Same silent-failure shape as the
finalize-without-publish bug. One-line import reorder in three files.
**Consequence for you: the base tasks must be re-registered (command below).**

**3. The publish reuse-branch read a status attribute that does not exist.**
`Dataset.status` is not a thing in clearml 2.1.10 — it always read as empty, so an
already-published dataset would be re-`publish()`ed (crash), and a dataset ABORTED
mid-build in the UI would have been blessed as published and trained on. Now reads
the real status from the backing task and stops dead on anything half-built.

## The bigger fixes behind those

| area | what was wrong | what it would have cost |
|---|---|---|
| clock measurement | clocks were measured AFTER the NaN policy ran — a 'zero' fill creates movement inside a bar → fast clock → **lookahead** | a leak, reintroduced through the side door |
| `na_policy: drop` | dropped the label minutes of the bad bar's STAMP; the poisoned minutes (where the bar is SERVED, one bar later) were kept | wrong rows deleted, poison kept |
| leak guard | ran its measured test only on files with a column literally named `close` — 103 of 285 v2 columns were silently never tested; id counters (`episode_id`, `pbsev_event_id`) passed | memorisation features in the dataset |
| span guard | compared raw timestamps, refusing legal joins at bar boundaries | false refusals |
| SHAP split | rebuilt from TODAY'S config, not the config the model trained under | explaining rows the model trained on = every SHAP number silently wrong. The split now travels INSIDE the model bundle |
| champion | a subset publish (`--models xgboost`) polled all three trainers forever and could never crown | judge waiting for runners who never entered |
| `local_check` | fed sentinel-filled data to xgboost/catboost (production keeps NaN) and integers to catboost (production feeds text) | the offline verdict measured a pipeline that does not exist |
| version bytes | re-publishing a version whose parquet was rebuilt silently reused the OLD bytes in ClearML | "v2" meaning two different datasets |
| `dataset_diff --all` | crashed on the first lock file or minor version | the ablation tool dead on first real use |
| requirements | matplotlib missing (every SHAP task dies on a fresh agent), pandas major unpinned (a pandas-2 machine is refused by its own certificate), 200 MB of dead notebook deps | broken fresh installs |
| lineage | `labels_name` was a hand-typed constant — v1 and v2 both claimed the OLD labels while their sha256 proved otherwise | a certificate that lies. Now derived from the file name |

## The tests themselves were part of the problem

- The no-peek test was **vacuous**: a pandas-3 unit change made it decode every bar
  into 1970, so "bar closed before the minute" was true for every row no matter what
  the aligner did. Green while proving nothing. Rebuilt unit-proof, with a non-vacuity
  assertion.
- The thin-evidence clock test failed the detector **for being right** (its fixture
  accidentally contained hard proof of a 1-min clock).
- `test_build_dataset.py` never imported `build_dataset.py`.
- New: `test_end_to_end_mini.py` runs the ENTIRE chain on a synthetic world in ~4s —
  register catches planted leaks, freeze applies the clock rules, build is certified,
  the certificate is verified against the parquet footer, the drop lands on the served
  window, no row receives an unclosed bar.

## Questions for the feature team (with evidence)

1. **`gap_state` and `volatility_scale` change once per session, at 09:15.** The
   measurement can only call them 60-min columns, so they are served an hour late and
   blanked for the first hour of every session. If they are computed from the OPEN
   (yesterday's close vs today's open), say so — one `clock_override` line each in
   registry.yaml makes them live at 09:16.
2. **The `*__mom` family is byte-identical to the `*__ema` family** (17 duplicate
   columns, ~6% of the dataset). That looks like an upstream compute bug, not a design
   choice. Which one is real?
3. **12 columns are >50% NaN** (worst: `window_oi_ratio` 95.5%, `frozen_high/low`
   93.9%, `pbsev_locked` 76.7%). Is NaN "no event" (fine, sentinel policy) or "not
   computed" (a problem)?
4. **NO_TRADE at `weight_raw` 7000 = 12.4% of the loss.** Target was 20–40%
   (`weight_raw` 12,000–33,000). Run `local_check` and read "minutes it would trade"
   before arguing.

**My Call:** the pipeline's ideas were right and its skeleton held; what failed was
everything at the JOINTS — import order, attribute names, unit conventions, config
drift. Every one of those was invisible to a per-file review and got caught only by
running the chain end to end. That end-to-end run is now a permanent 4-second test,
so this class of bug cannot ship silently again.
